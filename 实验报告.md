# 基于 Qt 的局域网通信软件实验报告

## 1. 概述

本次实验所分析和实现的是一个基于 Qt 的局域网通信软件。该系统面向同一局域网内的多台主机，目标是在不依赖中心服务器平台的前提下，实现用户之间的文字通信和视频通话功能，并在此基础上进一步支持群聊、文件传输、历史记录保存以及音频通信等扩展能力。与传统的互联网即时通信软件相比，局域网通信系统的运行环境更加封闭，网络延迟相对较低，主机之间的发现和连接可以通过广播与点对点方式直接完成，因此非常适合作为网络程序设计和 Qt 综合开发训练的实验项目。

从工程结构来看，该项目使用 C++ 作为开发语言，使用 Qt 6 作为主要开发框架，界面层基于 Qt Widgets 实现，网络部分主要依赖 Qt Network 模块，音频输入输出依赖 Qt Multimedia，而视频采集部分则结合 OpenCV 完成。工程中既包含界面逻辑，也包含通信协议、连接维护、媒体采集与显示等内容，属于一个具有较强综合性的桌面应用实验系统。程序运行后，用户可以自动发现局域网中的其他在线节点，与指定联系人进行单独聊天，也可以创建群聊进行多人文字交流；此外，系统还能够发起视频通话，通过摄像头采集本地图像，并将视频数据发送给对端显示。

本实验的意义在于，通过对该工程的分析和整理，可以系统理解 Qt 应用程序在实际通信场景中的综合使用方式。一方面，可以掌握界面设计、信号与槽机制、多线程调度等 Qt 开发中的核心方法；另一方面，也能够理解 UDP 广播发现、TCP 可靠传输、媒体数据处理与状态持久化等通信系统中的关键技术。通过本次实验，不仅能够加深对单个知识点的理解，更能体会它们如何共同支撑一个完整的软件系统。

## 2. 系统核心功能设计

### 2.1 基础连接功能

系统的基础连接功能是整个软件运行的前提。该项目虽然在整体上属于去中心化局域网通信程序，但从单个节点的运行方式来看，每个客户端实例在启动后都会同时具备“监听端”和“发起连接端”两种角色。程序启动时，会先在本地打开一个 TCP 监听端口，用于接收其他节点发起的连接请求；与此同时，还会启动基于 UDP 的局域网节点发现机制，通过固定端口向广播地址发送自身的节点编号、显示名称和本地监听端口。其他在线节点接收到这些广播后，就能够获知当前节点的网络地址和可连接端口，从而建立后续点对点会话。

在具体实现中，工程通过 `DiscoveryService` 负责广播发现逻辑，通过 `QTcpServer` 负责监听连接请求，通过 `PeerManager` 对所有已发现联系人和已建立连接进行统一管理。对于用户而言，这一过程是透明的，程序会自动在联系人列表中显示局域网中的其他节点，并在需要发送消息或发起通话时自动建立 TCP 连接。这样的设计既避免了手动输入大量网络参数的繁琐过程，也使系统在实验场景下具备较好的易用性。

### 2.2 文字通信功能

文字通信功能是本系统最基础、也是最核心的应用场景之一。工程在文字通信部分支持单聊和群聊两种模式。单聊模式下，用户可以在联系人列表中选择某一在线节点，在输入框中输入消息内容后直接发送。程序会将发送者身份、发送时间和消息正文封装为统一协议数据，通过 TCP 连接发送给目标用户。对端收到消息后，系统会立即在对应的会话窗口中显示内容，并更新本地聊天记录。

在群聊模式下，系统允许用户选择多个联系人创建一个群组，并自动为群组分配唯一编号。在群组建立之后，群消息的发送逻辑与单聊类似，但消息会根据群成员列表逐个转发给群内的其他节点。虽然这种做法与真正的服务器集中转发机制不同，但在局域网小规模实验场景中已经足够满足多人交流需求。除此之外，系统还实现了消息历史保存和未读消息计数功能，使文字通信不再只是即时显示，而具有较完整的桌面聊天软件特征。

### 2.3 视频通话功能

视频通话功能是该系统中技术含量较高的部分，也是本实验的重要特色。系统的视频通话建立在已有点对点连接基础之上，当用户在联系人列表中选择目标节点并发起通话请求后，程序会向对方发送呼叫邀请消息；对方若接受邀请，双方即进入通话状态，界面切换到通话页面，开始本地视频采集与远端视频播放。

在本地视频采集方面，系统使用 OpenCV 从摄像头中持续读取图像帧，并将这些图像转换为 Qt 可处理的 `QImage` 对象。考虑到原始视频数据量较大，程序不会直接把采集到的每一帧原图发送出去，而是先对其进行尺寸归一化和 JPEG 压缩，然后再通过 TCP 发送给对端。接收方收到压缩后的视频帧后，会先进行格式校验，再在独立线程中解码还原，最后显示在远端视频区域中。工程中虽然还实现了音频收发，使系统实际上具备音视频通话能力，但从本实验大纲出发，视频链路仍然是此模块分析的重点。总体来看，视频通话功能的设计较完整地体现了摄像头采集、媒体数据压缩、网络传输与界面播放之间的联动关系。

如图所示，系统总体模块图主要反映了本项目在结构层面的组成关系。整个系统以 `PeerManager` 为核心控制模块，上层连接用户界面，下层连接网络通信、音视频处理以及数据存储等功能模块，共同构成一个层次较为清晰的局域网通信软件。其中，界面层主要负责聊天界面、通话界面以及各类交互操作；网络通信层主要承担 UDP 节点发现、TCP 连接建立和协议解析等任务；音视频处理层主要完成摄像头采集、视频编解码和音频收发；数据与存储层则负责联系人信息、群聊信息、历史记录以及本地配置的维护。通过该图可以更加直观地看出，各模块之间并非彼此孤立，而是在核心控制模块的组织下形成统一协作关系，从而支撑系统完成文字通信和视频通话等主要功能。

如图所示，项目总体流程图主要描述了系统从启动到进入具体业务处理的整体运行过程。程序启动后，首先完成主窗口初始化和本地配置加载，随后启动 TCP 监听服务、UDP 节点发现服务以及相关线程资源。在成功发现局域网中的其他节点并更新联系人列表后，系统便可根据用户的操作进入不同的业务分支，包括文字聊天、群聊通信、文件传输以及视频通话等。由此可见，本系统的运行过程具有较为明确的层次性，即先建立底层通信环境和资源支撑，再逐步向上承载实际应用功能。总体流程图的作用在于帮助读者从动态过程角度把握整个系统的运行逻辑，从而更好地理解各功能模块在实际执行中的先后关系与协同方式。

## 3. 系统实现过程

### 3.1 服务器端实现

从传统意义上说，该工程并不存在一个独立部署的中心服务器，但若从功能职责上分析，每一个运行中的程序实例都承担了“本地服务器端”的一部分任务，因此完全可以从“服务器端实现”的角度理解其监听和转发机制。程序启动后，会在本地创建 `QTcpServer` 对象并监听一个 TCP 端口，该端口用于接收其他节点发起的连接请求。与此同时，UDP 广播服务会周期性地向局域网中发送本机信息，使其他节点能够感知到当前实例的存在。

当某个远端节点尝试连接本机时，本地监听端会接受这一连接，并为其封装出一个 `PeerConnection` 对象。该对象负责底层套接字的读写，并把接收到的原始字节流交给协议层进行拆包处理。进一步地，所有解析出的消息都会交由 `PeerManager` 统一分发。如果收到的是普通文本消息，则转交给界面层刷新聊天记录；如果收到的是群聊消息，则根据群编号分发到对应会话；如果收到的是文件开始、文件块或文件结束标记，则进入文件接收流程；如果收到的是视频帧或通话控制消息，则进一步转入媒体处理逻辑。由此可见，所谓“服务器端实现”，在该工程中主要表现为本地监听、连接接收、数据解析以及消息转发分发等职责。

### 3.2 客户端实现

客户端实现主要体现在图形界面构建和用户交互逻辑上。程序的主窗口采用 Qt Widgets 实现，界面整体分为联系人区域、群聊区域、聊天页面和通话页面几部分。用户可以在界面中设置自己的显示名称，查看局域网中已发现的联系人，选择某个联系人进行单聊，也可以创建群聊并切换到不同的群会话中进行消息交流。消息输入框支持直接输入文本并发送，同时聊天页面还提供文件发送与下载目录设置功能。在交互体验方面，程序还支持 `Enter` 快速发送消息、拖拽文件到聊天区域发送、未读消息数提示等设计，使系统更接近实际桌面聊天软件的使用习惯。

客户端在文字通信逻辑上主要负责读取用户输入、调用管理器发送消息、接收网络层返回的会话信号并更新聊天记录。在视频通信逻辑上，客户端不仅负责发起呼叫、接受呼叫和结束呼叫，还负责本地摄像头的配置、采集画面的预览、本地视频发送状态的切换以及远端视频区域的显示更新。由于视频压缩与解码开销较大，程序并未将这些操作直接堆积在界面线程中，而是借助独立线程完成编码和解码，从而保持客户端界面的响应能力。可以说，客户端部分不仅承担了“显示”的职责，也承担了通信行为发起、状态反馈和多媒体设备协调等重要功能。

### 3.3 关键技术要点

该项目在实现过程中涉及若干较为关键的技术点，分别体现在 Qt 信号与槽机制、TCP/UDP 协议协同使用以及多线程处理三个方面。首先，在 Qt 应用开发中，信号与槽机制是实现模块解耦的重要基础。本工程中，网络层在接收到文本消息、文件消息、通话状态或视频帧之后，并不会直接操作界面控件，而是通过信号将相关数据发送给主窗口，再由主窗口执行界面更新。这种方式使界面逻辑和网络逻辑保持了清晰边界，也有利于后续维护。

其次，项目合理地使用了 UDP 和 TCP 两种协议。UDP 主要用于节点发现，因为它支持广播发送，适合在局域网中快速通知其他主机自身的存在；TCP 则主要用于正式业务数据传输，包括文字消息、群消息、文件块、通话控制消息以及视频帧等。这样做的好处是，系统既获得了 UDP 发现效率高、部署简单的优点，又保留了 TCP 可靠传输、顺序保证的特性。

最后，多线程技术在本工程中具有十分重要的作用。摄像头采集、视频编码和视频解码都属于相对耗时的操作，如果全部在主线程中完成，很容易导致界面冻结或响应迟缓。因此，项目中专门设计了摄像头线程、视频编码工作线程和视频解码工作线程，使主界面线程只负责交互和显示，而将计算量较大的媒体处理交给后台线程完成。通过这种方式，系统在实现实时视频功能的同时，仍能够保持较好的界面流畅度。

## 4. 系统测试

为了验证该系统的功能完整性和运行稳定性，需要在局域网环境中进行较为全面的测试。测试环境通常包括两台或多台处于同一局域网的计算机，每台计算机均安装可运行 Qt 程序所需的依赖环境。若进行视频通话测试，则还需确保测试设备具有可正常使用的摄像头，并且 OpenCV 依赖已经正确配置。若同时测试音频能力，则还需要麦克风和扬声器等设备支持。

在核心功能测试中，首先需要验证节点发现与基础连接功能。测试方法是分别在不同主机上启动程序，观察联系人列表中能否自动出现对端节点；同时也可以点击“手动发现”按钮，验证系统能否立即刷新局域网节点状态。从源码逻辑看，程序通过 UDP 广播和应答机制完成节点识别，并通过 TCP 建立正式连接，因此在正常网络环境下，应能够较快完成发现和连接建立。其次，需要测试文字通信功能，包括单聊和群聊。单聊测试中，在一端发送消息后，另一端应能够及时显示收到的文本内容和发送时间；群聊测试中，应验证多个成员是否都能在群组会话中收到消息。根据工程实现逻辑，这些功能链路已经较为完整。

视频通话测试则是系统中最关键的一项。测试时，一方发起通话请求，另一方接受后，双方应进入通话页面，并在设备正常的前提下看到本地和远端的视频显示状态变化。需要观察摄像头是否能够被正确打开，视频采集是否稳定，远端图像是否能够被成功接收并显示出来。此外，若程序支持暂停本地视频发送，也应检查此状态是否能同步通知对方。由于该工程中视频数据需要经过采集、压缩、传输、解码和显示多个环节，因此该部分测试能够较全面反映系统整体协同效果。

在异常测试方面，应重点检查连接中断、文件传输不完整、摄像头打开失败以及节点离线等情况。比如，在文字消息发送或文件传输过程中断开网络连接，应观察程序是否能够及时提示通信失败；在接收文件时若数据不完整，系统是否会删除损坏文件；在摄像头不可用时，是否能够给出明确的错误提示而不是直接导致程序崩溃。结合源码实现来看，项目中已经加入了文件完整性校验、设备错误提示和连接关闭后的状态更新逻辑，因此在异常处理方面具备一定基础。综合测试结果分析，可以认为该系统已经较好地实现了局域网通信软件的主要实验目标，但在高并发、复杂网络环境以及性能优化方面仍有继续提升空间。

## 5. 开发中遇到的问题及解决方法

在该类局域网通信软件的开发过程中，首先遇到的一个关键问题是如何在没有中心服务器的条件下让不同节点自动发现彼此。若完全依靠用户手动输入对方 IP 地址和端口，不仅使用不便，也不符合局域网即时通信系统应有的特征。为解决这一问题，工程中采用了基于 UDP 广播的节点发现方法，通过周期性广播本机信息并在接收到他人公告后主动回复，使不同节点能够在局域网中快速完成彼此识别。这一方法较好地解决了节点初始化接入的问题。

第二个较为突出的难点是 TCP 通信中的数据边界处理问题。由于 TCP 属于字节流协议，并不天然区分消息边界，因此若直接按“接收到多少就处理多少”的方式编写程序，很容易出现粘包、拆包甚至消息解析混乱的情况。为解决这一问题，项目中专门设计了统一的数据包格式，在每个消息体前加入固定长度字段作为包头，再在包体内部存储消息类型、JSON 元数据和二进制数据。这样一来，接收端就能够根据长度字段准确拆分完整数据包，从而有效提高了通信的稳定性和协议一致性。

第三个问题主要出现在视频功能实现过程中。摄像头采集和视频压缩本身属于较耗时的操作，如果直接在主线程中执行，界面就会出现明显卡顿，严重时甚至会失去响应。为解决这一问题，开发过程中采用了多线程方案，将摄像头采集、视频编码和视频解码分别放入独立线程中执行，主线程只负责用户交互和界面显示。与此同时，系统还对视频帧进行了尺寸归一化和 JPEG 压缩，以减小网络传输负担，提高视频传输成功率。通过这种方式，视频通话功能在实验环境下能够达到较好的基本效果。

另一个开发中容易遇到的问题是异常情况下的资源处理。例如，在文件传输中，如果网络突然中断，可能导致对端只收到部分文件数据；如果不加处理，系统就会在本地留下损坏文件，影响使用体验。为解决这一问题，工程中在接收结束时增加了文件大小校验逻辑，只有当接收字节数与声明大小一致时才保留文件，否则自动删除并提示用户。再如，在摄像头设备不可用或打开失败时，系统会通过错误提示机制告知用户当前问题，而不是让程序无反馈地停止工作。这些问题的解决方法虽然属于细节处理，但正是它们提高了系统整体的可靠性和完整性。

## 6. 总结与体会

通过本次实验，对基于 Qt 的局域网通信软件有了更加全面和深入的认识。与以往单独练习某一个控件或某一种网络协议不同，这一项目把界面设计、网络通信、多线程处理、文件管理和视频采集播放等多项技术真正整合到了同一个工程之中，使人更加直观地体会到一个完整软件系统是如何从多个技术模块协同构成的。从这个意义上说，本实验最大的收获不仅是掌握了某个具体函数或类的使用方法，更重要的是理解了系统设计、模块划分和功能协同的重要性。

在完成本项目分析之后，可以明显感受到 Qt 在桌面通信软件开发中的优势。其信号与槽机制非常适合处理界面事件与网络事件之间的交互，Qt Network 为套接字通信提供了较好的封装，Qt Multimedia 则使音频采集和播放能够较方便地实现。再配合 OpenCV，便可以在桌面程序中较顺利地扩展出视频采集与显示能力。由此可见，Qt 不仅适合完成传统桌面界面开发，也能够胜任一定复杂度的多媒体通信应用原型开发。

当然，该系统仍然存在一些不足。首先，在架构层面，它虽然能够实现局域网内的直接通信，但并不适合更大规模、更复杂网络环境下的部署；其次，在媒体传输层面，视频当前采用 JPEG 帧压缩方式，音频采用较直接的数据传输方式，因此在带宽利用率和实时性方面仍有提升空间；再次，在安全性方面，系统尚未加入身份认证、权限控制和加密机制，因此更适合作为实验原型而非正式产品。后续若要进一步优化，可以考虑引入更高效的视频编解码方式、改进音视频传输协议、增强异常恢复机制，并在系统安全层面进行补充。

总体来看，本次实验较好地完成了局域网通信软件分析与总结任务。该工程实现了文字通信、群聊、视频通话等主要功能，结构较为清晰，功能链路较完整，具有较高的综合实验价值。通过本次实验，不仅巩固了对 Qt 编程、网络通信和多线程技术的理解，也进一步提升了从项目工程出发进行系统分析和书面总结的能力，这对于后续继续开展更复杂的软件系统开发具有重要帮助。
